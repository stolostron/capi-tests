# Scheduled Repository Review

Automatically review all new GitHub issues and new/updated pull requests from the last 24 hours.
Generate a summary report, analyze each item, and create draft PRs where appropriate.

## Step 1: Discover new issues (last 24h)

Find all issues created in the last 24 hours:

```bash
gh issue list --repo stolostron/capi-tests --state open \
  --json number,title,body,labels,createdAt,author \
  --jq '[.[] | select(.createdAt > (now - 86400 | strftime("%Y-%m-%dT%H:%M:%SZ")))]'
```

If there are no new issues, note "No new issues in the last 24 hours" and move to Step 2.

### For each new issue:

1. **Analyze impact**:
   - Read the issue title and body carefully
   - Identify which files, test phases, or components are affected
   - Classify severity: critical (blocks testing), moderate (degrades functionality), low (cosmetic/docs)
   - Check if there are related issues or PRs already

2. **Create a Draft PR with implementation suggestion**:
   - Create a branch: `auto/issue-<number>-<short-description>` (from latest `main`)
   - Analyze the codebase to understand what changes are needed
   - Implement the suggested fix following CLAUDE.md patterns
   - Run `make fmt` and `make lint` to ensure code quality
   - Commit with message: `fix: <description> (addresses #<number>)`
   - Push to origin and create a Draft PR against `upstream` (stolostron/capi-tests):
     ```bash
     gh pr create --repo stolostron/capi-tests --draft \
       --title "fix: <description> (addresses #<number>)" \
       --body "## Description\n\nAutomated implementation suggestion for #<number>\n\n## Changes Made\n\n- <changes>\n\n## Additional Notes\n\nThis is an automated draft PR. Please review before merging.\n\n🤖 Generated by Scheduled Review Assistant"
     ```
   - If implementation is too complex or ambiguous, skip the PR and note it in the report with reasoning

3. **Record for summary**: issue number, title, severity, PR number (if created), or reason skipped

## Step 2: Discover new/updated PRs (last 24h)

Find all open PRs updated in the last 24 hours (excluding draft PRs created by this workflow):

```bash
gh pr list --repo stolostron/capi-tests --state open \
  --json number,title,body,updatedAt,author,isDraft,headRefName \
  --jq '[.[] | select(.updatedAt > (now - 86400 | strftime("%Y-%m-%dT%H:%M:%SZ"))) | select(.headRefName | startswith("auto/") | not)]'
```

If there are no updated PRs, note "No new/updated PRs in the last 24 hours" and move to Step 3.

### For each PR:

1. **Run comprehensive review**:
   - Use the `pr-review-toolkit:review-pr` agent to analyze the PR
   - Collect all findings (bugs, style issues, security concerns, test gaps)

2. **If findings exist, create a Draft PR with fixes**:
   - Create a branch: `auto/review-pr-<number>` (from the PR's head branch)
   - For each finding, make the fix in a **separate commit**:
     - Commit message: `fix(<scope>): <finding description> (review of PR #<number>)`
   - Run `make fmt` and `make lint`
   - Push to origin and create a Draft PR against the **original PR's head branch**:
     ```bash
     gh pr create --repo stolostron/capi-tests --draft \
       --base <pr-head-branch> \
       --title "review: suggested fixes for PR #<number>" \
       --body "## Description\n\nAutomated review findings for #<number>\n\nEach commit addresses a separate finding.\n\n## Changes Made\n\n- <list of findings and fixes>\n\n## Additional Notes\n\nThis is an automated review. Please evaluate each commit independently.\n\n🤖 Generated by Scheduled Review Assistant"
     ```
   - If no actionable findings, note "PR #<number> looks good, no issues found"

3. **Record for summary**: PR number, title, findings count, fix PR number (if created)

## Step 3: Generate summary report

Print a markdown summary to stdout:

```markdown
# 📋 Daily Repository Review — <date>

## New Issues (last 24h)

| # | Title | Severity | Action | Draft PR |
|---|-------|----------|--------|----------|
| <number> | <title> | <severity> | <implemented/skipped> | #<pr> or N/A |

## Updated PRs (last 24h)

| # | Title | Findings | Action | Fix PR |
|---|-------|----------|--------|--------|
| <number> | <title> | <count> | <fixes created/clean> | #<pr> or N/A |

## Statistics
- Issues reviewed: <count>
- PRs reviewed: <count>
- Draft PRs created: <count>
- Total findings: <count>
```

Also write this summary to `$GITHUB_STEP_SUMMARY` if running in GitHub Actions.

## Important Guidelines

- **Never force-push** or modify existing branches
- **Always create Draft PRs** — never ready-for-review PRs
- **Skip auto/* branches** when reviewing PRs (avoid reviewing our own automated PRs)
- **Follow CLAUDE.md** patterns for all code changes
- **One commit per finding** in review PRs for easy cherry-picking
- **Be conservative**: skip implementation if the issue is ambiguous or requires design decisions
- If `make lint` or `make fmt` fails, fix the issues before committing
- Use `Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>` in all commits
